## What Is This?

This is the authoritative field-by-field reference for the Add Value Template form in AI Control Tower. Where Topic 21 (Using Value Templates) explained the concepts and procedures, this reference document is the definitive specification of every field, its meaning, its constraints, and how it affects the value calculation output.

Use this reference when building new value templates, troubleshooting calculation discrepancies, reviewing templates for accuracy during the Value Template review task in the Assess lifecycle stage, or designing custom metrics.

---

## Form Overview

The Add Value Template form is organized into three sections:
1. **Calculation builder** — defines the formula components
2. **Attributes** — defines the template metadata
3. **Calculation output** — defines the instance scope

The formula generated by this form produces a **daily value in minutes** for each AI asset the template is mapped to.

**Core formula structure:**
```
Value (minutes/day) = Usage × Time × (Acceptance Rate / 100)
```

Each component (Usage, Time, Acceptance Rate) can be either a **constant** (fixed value) or an **indicator** (dynamic metric derived from a custom performance analytics indicator).

## Section 1: Calculation Builder

### Persona

| Attribute | Specification |
|---|---|
| **Field name** | Persona |
| **Options** | Agent, Developer, Other, Requestor |
| **Default** | Other |
| **Purpose** | Scopes the value calculation to a specific user role |
| **Effect** | Value calculations per this template apply only to the selected persona |

**Example:** If the AI asset is Incident Summarization and the selected persona is Agent, the value calculations are for the Agent role only — not for Requestors or Developers who may also use the skill.

> **Governance note:** Persona scoping allows differentiated value measurement for the same AI asset across different user populations. An AI system used by both agents and developers may deliver different time savings per execution for each group. Create separate templates per persona if you need role-differentiated value reporting.

---

### Usage

| Attribute | Specification |
|---|---|
| **Field name** | Usage |
| **Purpose** | The usage metric of the AI asset used to calculate value |
| **Type constraint** | Must be a **daily usage indicator** |
| **Custom metric** | If you add a custom indicator, the usage value is derived from the indicator |

**Example:** For Incident Summarization, the usage value is the daily executions of Incident Summarization.

> **Governance note:** The "daily" constraint is critical. If you use a non-daily indicator, the calculation will produce incorrect results because the formula multiplies Usage × Time × Acceptance Rate on a daily basis. Always confirm the indicator aggregation period is daily before mapping to Usage.

---

### Time Value Type

| Attribute | Specification |
|---|---|
| **Field name** | Time value type |
| **Options** | Constant, Indicator |
| **Purpose** | Determines how the time component of the formula is sourced |
| **When to use Constant** | When you have a known, fixed time saved per execution |
| **When to use Indicator** | When you have a custom metric measuring actual time saved |

---

### Time Constant

| Attribute | Specification |
|---|---|
| **Field name** | Time constant |
| **Unit** | Minutes |
| **Purpose** | A fixed time value used as the baseline time component in the calculation |
| **Active when** | Time value type = Constant |

**Example:** For Incident Summarization, the time constant is the average time saved per invocation of Incident Summarization — for example, 0.2 minutes (12 seconds) per execution.

> **Governance note:** The Time constant is the most commonly misspecified field in value templates. It should represent the time saved per execution, not the time the AI takes to run. "Time saved" means: how much human time would this task have taken without the AI, minus any human time still required with the AI. During the Value Template review task, verify this number is evidence-based (e.g., from time studies or user surveys), not assumed.

---

### Time Indicator

| Attribute | Specification |
|---|---|
| **Field name** | Time indicator |
| **Unit** | Minutes per invocation (derived from custom indicator) |
| **Purpose** | The average time saved per invocation derived from a custom performance analytics indicator |
| **Active when** | Time value type = Indicator |

**Example:** If the AI asset is Incident Summarization, the time indicator is the average time saved per invocation, derived from a custom indicator that measures actual agent time-on-task before and after AI assistance.

> **Governance note:** Using an Indicator for Time is more accurate than a Constant because it reflects actual measured performance rather than an assumption. For high-value AI systems, invest in building proper performance analytics indicators for Time. The extra setup cost pays off in credible ROI reporting.

---

### Acceptance Rate Type

| Attribute | Specification |
|---|---|
| **Field name** | Acceptance rate type |
| **Options** | Constant, Indicator |
| **Purpose** | Determines how the acceptance rate component of the formula is sourced |
| **When to use Constant** | When using a target or estimated acceptance rate |
| **When to use Indicator** | When you have a custom metric measuring actual acceptance |

---

### Acceptance Rate Constant

| Attribute | Specification |
|---|---|
| **Field name** | Acceptance rate constant |
| **Unit** | Percentage (1–100) |
| **Constraint** | Value must be between 1 and 100 |
| **Purpose** | A fixed percentage representing the expected or target acceptance rate |
| **Active when** | Acceptance rate type = Constant |

**Example:** For Incident Summarization, the acceptance rate constant is the percentage of Incident Summarization results that are acceptable to users — for example, 90% means 90 out of every 100 AI outputs are accepted and used.

> **Governance note:** The default acceptance rate for most templates is 50%. This is a conservative assumption meaning only half of AI outputs are accepted. If your actual acceptance data shows higher rates, the default template understates value. If lower, it overstates. The Value Template review task exists precisely to validate this number. A 50% acceptance rate with no measurement evidence is a governance gap — it is an untested assumption that could materially misrepresent the AI program's value to executives.

---

### Acceptance Rate Indicator

| Attribute | Specification |
|---|---|
| **Field name** | Acceptance rate indicator |
| **Unit** | Percentage (1–100) per day |
| **Constraints** | Must be a **daily indicator** with value between 1 and 100 |
| **Purpose** | Dynamic acceptance rate derived from a custom performance analytics indicator |
| **Active when** | Acceptance rate type = Indicator |

**Example:** For Incident Summarization, the acceptance rate indicator is the percentage of Incident Summarization results that were accepted by users — derived from a custom indicator that tracks user acceptance actions (e.g., "copied to clipboard," "applied to ticket") as a ratio of total executions.

> **Governance note:** Like the Time indicator, using an actual measured acceptance rate indicator is significantly more credible than a constant. This is especially important for AI systems used to justify significant investment — an evidence-based acceptance rate that can be audited is far stronger governance documentation than an assumed 50%.

## Section 3: Calculation Output

### Instance

| Attribute | Specification |
|---|---|
| **Field name** | Instance |
| **Purpose** | Specifies which instance the value template is applied to (useful for multi-instance management) |
| **Testing note** | You can validate a formula in a non-production instance before publishing to production |
| **Testing timing** | Same-instance testing: up to 10 minutes wait. Cross-instance testing: up to 30 minutes wait |
| **During testing** | Cannot edit or resend the template for testing; can continue working with other templates |

## The Formula in Full

For a template with all constant values, the full calculation is:

```
Daily Value (minutes) = 
  [Daily executions per day (Usage)] 
  × [Average time saved per execution in minutes (Time constant)]  
  × [Acceptance rate percentage (Acceptance rate constant) / 100]
```

For a template using all indicator values:

```
Daily Value (minutes) = 
  [Custom daily usage indicator value]
  × [Custom daily time indicator value in minutes]
  × [Custom daily acceptance rate indicator value / 100]
```

Mixed configurations (constant Time + indicator Acceptance rate, etc.) are valid.

## Key Takeaways for AI Risk & Compliance Officers

1. **Every component can be constant or indicator** — constants are assumptions; indicators are measurements; governance programs should migrate from constants to indicators as data becomes available.
2. **Usage must be daily** — non-daily indicators will produce wrong calculations; verify aggregation period before publishing.
3. **Acceptance rate constant default of 50% is an assumption, not a measurement** — the Value Template review task must validate this with evidence.
4. **Time constant = time saved, not time taken** — the most common error; human time delta before vs. after AI, not AI execution time.
5. **Publishing updates ALL instances** — test in non-production; allow up to 30 minutes for cross-instance testing; cannot edit during testing window.
6. **Template name must be unique** — establish a naming convention before creating templates at scale.
7. **Persona scoping enables role-differentiated value reporting** — use it when different user groups derive different value from the same AI system.
