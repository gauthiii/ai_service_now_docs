# GRC: Entity Based Access for AI Assets

## Overview

The **GRC: Entity Based Access** application enables you to segregate data on the AI asset records to ensure that only authorized users can access sensitive AI Risk and Compliance data, while still maintaining visibility into core entities.

Entity-based access administrators can use this application to set up secure, controlled access to various AI assets and their related objects.

## What Is Entity-Based Access (EBA)?

**Entity-Based Access (EBA)** is a security feature designed to provide **granular, data-level access control** within the AI Risk and Compliance application.

Unlike **role-based access control**, EBA decides which records a user can access based on **business entities**, such as:

- Departments
- Regions
- Business units

This approach ensures that sensitive information is only accessible to authorized users, aligning with organizational compliance and confidentiality requirements.

### Key Distinction

| Access Model | Basis for Access Decisions |
|---|---|
| Role-Based Access Control | A user's assigned roles |
| Entity-Based Access (EBA) | Business entities (departments, regions, business units) tied to records |

## What Data Can Be Controlled

AI Risk and Compliance managers can access the following data through entity-based access:

- Risks
- Controls
- Related entities
- Issues
- Indicators
- AI asset tasks
- Risk assessments
- Attestations
- AI assets data

### Visibility Behavior

- **Entities themselves stay visible to all users.**
- **Visibility of linked records is limited to authorized users.**

## Tables Accessible to Qualified Users

When a user is qualified based on the EBA configurations **and** has the **minimum required roles**, they have access to the following tables:

| Table (Label) | Table Name |
|---|---|
| AI system | `sn_grc_ai_gov_ai_system` |
| AI system entity | `sn_grc_ai_gov_ai_system_entity_map` |
| AI system task | `sn_grc_ai_gov_ai_system_task` |

## Configuring GRC: Entity Based Access

The following tasks must be performed to enable and use GRC: Entity Based Access for the AI asset records.

1. **Install the GRC: Entity Based Access application.**
   - Refer to *Install the Entity Based Access application*.
2. **Enable or disable the entity-based Access properties** to control access to the objects that are associated with an AI asset.
   - Refer to *Set up Entity Based Access properties*.
3. **Configure an entity class for a linked object** by using the GRC: Entity Based Access application.
   - Refer to *Configure an entity class for a linked object*.
4. **Configure an entity type** by using the GRC: Entity Based Access application.
   - Refer to *Configure an entity type for a linked object*.
5. **Set access restrictions for the existing records in bulk** by using the entity-based record access update utility guided experience.
   - Refer to *Set access restrictions using an entity based record access update utility*.
6. **Configure entity-based record access rules on record types** to apply access restrictions to new records automatically.
   - Refer to *Configure entity-based record access rules*.

### Note: Entity Classes for AI Assets

Entities created with an AI asset are assigned an entity class such as:

- AI system
- AI model
- Dataset
- MCP server

The class assigned depends on the entity's category. To apply access restrictions to these entities, you must configure the appropriate entity class settings.

## Default Records and Field Configurations

Three records are provided by default, each with specific field configurations.

| Record (Table Name) | User Fields | Group Fields |
|---|---|---|
| AI Asset (`sn_grc_ai_gov_ai_system`) | Analyst, Business owner | Analyst Group |
| AI Asset task (`sn_grc_ai_gov_ai_system_task`) | Assigned to, Watch list | — |
| Related Entity (`sn_grc_ai_gov_ai_system_entity_map`) | None configured by default | None configured by default |

## Summary

GRC: Entity Based Access for AI assets brings **data-level, entity-driven access control** to the AI Risk and Compliance application. By restricting record visibility based on business entities rather than roles alone, it enforces confidentiality and compliance requirements while keeping core entities visible to all users. Implementation involves installing the application, configuring access properties, defining entity classes and types, applying bulk access restrictions to existing records, and setting up automatic access rules for new records.
