# Asset-Specific Parameter Mapping

> Section: Manual Intake Configuration – Record Producers and Workspace → Asset-Specific Parameter Mapping
> Source: AI-Control-Tower.pdf, p.57

---

## What Is This?

Asset-Specific Parameter Mapping defines how the parameterized Intake-form UI Builder page dynamically renders the correct intake form based on the AI asset type being created. Each intake action passes a specific `table` parameter value that determines which AI asset record is captured. This mapping is the link between an intake action (such as adding an AI System or Model) and the underlying digital asset table used to store its details.

---

## Asset-Specific Parameter Mapping

The Intake-form page is parameterized to dynamically render forms based on the AI asset type being created, using the `table` parameter to define the asset type and `sys_id` set to `-1` for new records. The following mapping defines the parameter value (table) used for each intake action.

| Action | Parameter Value (table) | Purpose |
|---|---|---|
| Add AI System | `alm_ai_system_digital_asset` | Captures AI System (use case) details |
| Add AI Model | `alm_ai_model_digital_asset` | Captures AI Model details |
| Add Dataset | `alm_ai_dataset_digital_asset` | Captures dataset details |
| Add Prompt | `alm_ai_prompt_digital_asset` | Captures prompt details |

---

## How to Use This in Implementation

When configuring manual intake actions, set the Intake-form page's `table` parameter to the corresponding digital asset table value for the asset type being created, and set `sys_id` to `-1` for new records. For example, an "Add AI System" action should pass `table=alm_ai_system_digital_asset`, while "Add Prompt" passes `table=alm_ai_prompt_digital_asset`. This ensures the reusable Intake-form UI Builder page renders the correct form and writes to the appropriate AI asset table for each intake type.
