## What Is This?

This subtopic describes how the reusable Intake-form UI Builder page is parameterized so it can dynamically render different intake forms depending on which AI asset type is being created. It explains the page parameters that drive this behavior and the asset-specific mapping that determines which table each intake action targets. It fits into the manual intake configuration that lets users add AI systems, models, datasets, and prompts through a single workspace page.

---

## Page Parameters and Dynamic Behavior

The Intake-form page is parameterized to dynamically render forms based on the AI asset type being created.

- Parameters:
  - `table` → Defines the asset type
  - `sys_id` → Set to `-1` for new records

---

## Asset-Specific Parameter Mapping

| Action | Parameter Value (table) | Purpose |
|---|---|---|
| Add AI System | `alm_ai_system_digital_asset` | Captures AI System (use case) details |
| Add AI Model | `alm_ai_model_digital_asset` | Captures AI Model details |
| Add Dataset | `alm_ai_dataset_digital_asset` | Captures dataset details |
| Add Prompt | `alm_ai_prompt_digital_asset` | Captures prompt details |

---

## How to Use This in Implementation

When configuring the manual intake experience, point each intake action (Add AI System, Add AI Model, Add Dataset, Add Prompt) at the single reusable Intake-form UI Builder page and pass the appropriate `table` parameter value from the mapping above. Set the `sys_id` parameter to `-1` so the page renders a blank form for a new record. Because one page serves all asset types, the correct `table` value is what drives which form is rendered and which table the record is created in, so the exact identifiers must be supplied verbatim.
