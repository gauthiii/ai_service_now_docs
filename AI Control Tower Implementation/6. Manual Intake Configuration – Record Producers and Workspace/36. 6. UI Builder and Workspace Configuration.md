## What Is This?

This subtopic provides best-practice guidance for configuring the manual intake user interface using UI Builder and the AI Control Tower Workspace. It focuses on maintaining a single, consistent, parameter-driven intake experience while preserving the out-of-the-box governance baseline. It fits within the manual intake configuration phase, where record producers and workspace forms are customized.

---

## UI Builder and Workspace Configuration Guidelines

- Maintain single intake experience using parameter-driven UI (`table` parameter)
- Use Intake view for all form customizations
- Avoid removing OOTB fields (minimum governance baseline)
- Validate all UI changes against:
  - Lifecycle workflows
  - Data model dependencies
- Carefully manage:
  - Client scripts (model table logic)
  - Stepper behavior (client state parameters)
- Restrict prompt creation to Workspace (not Service Catalog)

---

## How to Use This in Implementation

When customizing the AICT manual intake UI, work within UI Builder using the `table` parameter so a single parameter-driven intake experience is maintained across asset types. Apply all form customizations through the Intake view, and never remove out-of-the-box fields, since they constitute the minimum governance baseline. Before publishing changes, validate them against lifecycle workflows and data model dependencies, and take care when editing client scripts (which carry model table logic) and stepper behavior (driven by client state parameters). Finally, ensure prompt creation is confined to the Workspace rather than the Service Catalog.
