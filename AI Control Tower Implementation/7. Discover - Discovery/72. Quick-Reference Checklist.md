## What Is This?

This is an Appendix quick-reference checklist used when configuring the Discovery layer for AI Control Tower. It consolidates the key prerequisites, role assignments, asset definitions, and design best practices into a single pre-go-live verification list. Implementers use it as a final readiness gate before the Discovery layer goes live.

---

## Pre-Go-Live Discovery Checklist

Use the following checklist when configuring the Discovery layer for AI Control Tower.

| ☐ | Pre-Go-Live Discovery Checklist |
| --- | --- |
| ☐ | AI Control Tower SKU and sn_ent plugin are provisioned and active. |
| ☐ | Users requiring API access hold asset and model_manager roles. |
| ☐ | AI asset record types (System, Model, Dataset, Prompt) are defined and scoped. |
| ☐ | Record Producer forms are limited to required fields; UI policies replace client scripts where possible. |
| ☐ | Variable sets are configured for reuse across asset intake forms. |
| ☐ | Post-submission logic is routed to Flow Designer or Subflows (not inline producer scripts). |
| ☐ | UI Builder layouts use OOTB components; no unnecessary custom components. |
| ☐ | Data Resources are filtered — no wildcard queries in production. |
| ☐ | All UI Builder experiences tested with real user personas and role simulations. |
| ☐ | AWS Bedrock and Azure AI Foundry integration docs reviewed when available. |

---

## How to Use This in Implementation

Apply this checklist as the final verification step before bringing the Discovery layer live. Confirm platform prerequisites first (AI Control Tower SKU and the `sn_ent` plugin provisioned and active), then validate that users needing API access hold the `asset` and `model_manager` roles. Ensure all four AI asset record types (System, Model, Dataset, Prompt) are defined and scoped, and that intake form design follows best practices — minimal Record Producer fields, UI policies over client scripts, reusable variable sets, and post-submission logic routed through Flow Designer or Subflows rather than inline producer scripts. For UI Builder, use OOTB components, filter Data Resources to avoid wildcard queries in production, and test all experiences with real user personas and role simulations. Finally, review AWS Bedrock and Azure AI Foundry integration documentation when it becomes available.
