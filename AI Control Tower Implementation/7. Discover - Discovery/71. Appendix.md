## What Is This?

The Appendix for the Discover - Discovery section provides a consolidated Quick-Reference Checklist for configuring the Discovery layer of AI Control Tower. It is a pre-go-live verification aid that implementers use to confirm that prerequisites, roles, asset types, forms, and Discovery configurations are correctly set up before launch.

---

## Quick-Reference Checklist

Use the following checklist when configuring the Discovery layer for AI Control Tower.

| Pre-Go-Live Discovery Checklist |
| --- |
| ☐ AI Control Tower SKU and sn_ent plugin are provisioned and active. |
| ☐ Users requiring API access hold asset and model_manager roles. |
| ☐ AI asset record types (System, Model, Dataset, Prompt) are defined and scoped. |
| ☐ Record Producer forms are limited to required fields; UI policies replace client scripts where possible. |
| ☐ Variable sets are configured for reuse across asset intake forms. |
| ☐ Post-submission logic is routed to Flow Designer or Subflows (not inline producer scripts). |
| ☐ UI Builder layouts use OOTB components; no unnecessary custom components. |
| ☐ Data Resources are filtered — no wildcard queries in production. |
| ☐ All UI Builder experiences tested with real user personas and role simulations. |
| ☐ AWS Bedrock and Azure AI Foundry integration docs reviewed when available. |

---

## How to Use This in Implementation

An implementer applies this checklist as a final pre-go-live verification gate for the Discovery layer. Before launch, confirm each item: provisioning of the AI Control Tower SKU and `sn_ent` plugin; assignment of `asset` and `model_manager` roles for API access; definition and scoping of the AI asset record types (System, Model, Dataset, Prompt); proper form construction (minimal required fields, UI policies over client scripts, reusable variable sets); routing of post-submission logic through Flow Designer or Subflows rather than inline producer scripts; use of OOTB UI Builder components; filtered Data Resources with no wildcard production queries; persona/role-based testing of all UI Builder experiences; and review of AWS Bedrock and Azure AI Foundry integration docs when available.
