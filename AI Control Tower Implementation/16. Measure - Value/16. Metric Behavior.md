## What Is This?

This subtopic defines how each value **metric** behaves — where its data comes from, how it can be configured, and the ordering dependency it has on scheduled value calculations. It complements the *Metric Definitions* (Persona, Usage, Time Saved, Acceptance Rate) by describing the runtime characteristics of the three calculated metrics used in the productivity formula `Productivity = Usage × Time Saved per Invocation × Acceptance Rate`.

---

## Metric Behavior

- **Usage**
  - Derived from daily **PA (Performance Analytics) indicators**.
  - **Requires completion before scheduled value calculations** run.
- **Time Saved**
  - Can be:
    - **Static value** (e.g., 15 minutes)
    - **Indicator-based**
- **Acceptance Rate**
  - **Formula: Accepted Outputs ÷ Total Outputs**
  - Configurable as **static** or **indicator-driven**

---

## How to Use This in Implementation

When configuring value measurement, decide for each metric whether it should be **static** or **indicator-based**:

- **Usage** is always indicator-driven (daily PA indicators) — ensure the PA collection jobs complete **before** the scheduled value calculation jobs, otherwise productivity figures will be computed against incomplete usage data.
- **Time Saved** can be set to a fixed estimate (e.g. 15 minutes per invocation) for simplicity, or wired to an indicator for more accuracy.
- **Acceptance Rate** can likewise be a fixed percentage or derived from Accepted ÷ Total Outputs indicators.

The dependency note ("Usage requires completion before scheduled value calculations") is the key operational gotcha: schedule the value-calculation job to run after the daily PA indicator refresh.
