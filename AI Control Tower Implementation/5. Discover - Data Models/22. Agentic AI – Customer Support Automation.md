## What Is This?

This subtopic presents a worked example of how an Agentic AI use case maps onto the AICT data model. The scenario is a virtual agent that autonomously handles customer inquiries end-to-end. It illustrates how Agentic AI differs from Traditional AI by introducing multi-system orchestration and multi-model dependencies, which AICT must track at a more granular level.

---

## Scenario

**Scenario:** A virtual agent autonomously handles customer inquiries end-to-end.

## Data Model Mapping

| Component | Example |
|-----------|---------|
| Business Application | AI-powered Customer Support Agent |
| AI Type | Agentic AI |
| AI System | Customer Support Agent |
| Sub-Systems | Issue Summarization, Owner Assignment, Knowledge Retrieval |
| AI Models | LLM (summarization), classification model (urgency), retrieval model (KB lookup), dialogue model |
| Dataset | Historical tickets, chat logs, escalation cases, knowledge base |
| Input | Customer query, account metadata |
| Prompt | Instruction to resolve queries, summarize issues, retrieve KB content |
| Output | Issue summary, automated response, escalation when required |

### Important

> Agentic AI introduces multi-system orchestration and multi-model dependencies, requiring more granular tracking of relationships and execution flows within AICT.

---

## How to Use This in Implementation

When modeling an Agentic AI use case in AICT, capture the full hierarchy: the Business Application, the overarching AI System, its constituent Sub-Systems, and each underlying AI Model. Because Agentic AI orchestrates multiple sub-systems and depends on multiple models (e.g., LLM for summarization, classification for urgency, retrieval for KB lookup, and a dialogue model), implementers must record the relationships and execution flows between these components rather than treating the system as a single model. Datasets, inputs, prompts, and outputs should be documented per the example to support governance and traceability across the orchestrated flow.
