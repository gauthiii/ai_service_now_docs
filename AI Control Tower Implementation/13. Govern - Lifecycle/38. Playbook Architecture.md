## What Is This?

The playbook architecture describes the two core components that drive the AI asset governance lifecycle in AI Control Tower: the automation layer and the UI layout. It explains how lifecycle steps and tasks are presented in the workspace and how the underlying automation framework orchestrates lifecycle phases. This is where implementers understand how to extend and customize the lifecycle playbook while preserving upgrade compatibility.

---

## Playbook Components

The playbook consists of two core components.

### 1. Automation Layer

**UI Layout** defines how lifecycle steps and tasks are presented in the workspace. It is implemented as a list view of lifecycle tasks, including status and progression indicators.

The underlying automation framework is powered by a central subflow — the **AI Asset Lifecycle Assessment Flow** — which orchestrates lifecycle phases, manages task creation and progression, and ensures alignment between governance and execution layers. Each lifecycle phase has specific flows and subflows triggered based on conditions defined in Flow Designer.

### 2. UI Layout - Customization and Extensibility

Customers can extend the playbook through Flow Designer. Following is the underlying workflow design:

- **Clone, don't modify** — Clone existing Flows and Subflows and adjust triggers and conditions to support additional AI asset types. Avoid deactivating OOB tasks or modifying them directly to preserve upgrade compatibility.
- **Task customization** — Define new lifecycle tasks using Flow Designer conditions and adjust sequencing as needed.
- **Lifecycle stage order** — Maintain the standard Assess → Build & Test → Deploy sequence.

## How to Use This in Implementation

When extending the lifecycle playbook, implementers should work through Flow Designer rather than editing out-of-the-box artifacts directly. Clone existing Flows and Subflows to add support for new AI asset types, define new lifecycle tasks via Flow Designer conditions, and adjust task sequencing as needed — while always keeping the standard Assess → Build & Test → Deploy stage order. Avoiding deactivation or direct modification of OOB tasks preserves upgrade compatibility. The central AI Asset Lifecycle Assessment Flow subflow is the orchestration point that ties governance and execution together, so customizations should be designed to align with its phase-based, condition-triggered structure.
