## What Is This?

Access Control Refinement is a prelaunch task that tightens access control configuration beyond the initial role provisioning. Its goal is to enforce least-privilege access and protect sensitive governance data in AI Control Tower. It uses Access Control Lists (ACLs) at the field and table level to ensure each persona sees only what they should.

---

## Access Control Refinement

Refine access control configuration beyond initial role provisioning to enforce least-privilege access and protect sensitive governance data.

### Access Control Checklist

| ✓ | Access Control |
| --- | --- |
| ☐ | Verify all ACLs follow the principle of least privilege |
| ☐ | Use field-level ACLs for sensitive data; use table-level ACLs for general control |
| ☐ | Document any custom ACLs with purpose, roles, and conditions |
| ☐ | Test access for each persona type to confirm appropriate visibility and restrictions |
| ☐ | Avoid duplicating ACLs for the same operation |
| ☐ | Enforce data protection with server-side ACLs; do not rely on client-side scripts or policies |

---

## How to Use This in Implementation

After completing initial role provisioning, work through each checklist item to harden access. Apply field-level ACLs to sensitive governance data and table-level ACLs for broader control, ensuring every ACL adheres to least privilege. Document each custom ACL with its purpose, applicable roles, and conditions, and test access per persona type to confirm correct visibility and restrictions. Avoid duplicate ACLs covering the same operation, and always enforce protection server-side rather than relying on client-side scripts or policies. Custom ACLs documented and tested here feed directly into the later Pre-Go-Live Validation checklist.
