## What Is This?

AI assets in ServiceNow are deployed across three primary implementation patterns. Each pattern carries different implications for how Configuration Items (CIs) are discovered, registered, and related in the CMDB. Understanding the pattern an AI asset follows tells implementers how its CI will appear in inventory and how it relates to parent applications, platforms, and other agents.

---

## The Three AI Implementation Patterns

AI assets are deployed across three primary implementation patterns. Each pattern carries different implications for how CIs are discovered, registered, and related in the CMDB.

| Pattern | Examples | CMDB Implications |
|---------|----------|-------------------|
| Embedded AI | Now Assist skills in ServiceNow, AI features in CRM or ERP, intelligent document processing | AI system CI relates to the parent business application. Often auto-discovered (Now Assist) or submitted via manual intake (third-party). |
| Dedicated AI Platform | Amazon Bedrock, Azure AI Foundry, GCP Vertex AI, Copilot Studio | Platform is registered as a business applications. Individual models and agents are child CIs. OOTB connectors auto-discover. |
| Agent-to-Agent / MCP | Agents calling other agents via MCP, multi-agent orchestration, external tool access | Dependencies between agents are the primary governance concern. Capture agent topology in Enterprise Architecture. MCP is a gateway/integration technology; align governance with API Insights. |

---

## How to Use This in Implementation

Use the pattern to determine how each AI asset enters and relates within the CMDB:

- For **Embedded AI**, expect the AI system CI to relate to its parent business application. Now Assist assets are typically auto-discovered, while third-party embedded AI is captured via manual intake.
- For a **Dedicated AI Platform**, register the platform as a business application and model individual models and agents as child CIs; rely on out-of-the-box (OOTB) connectors for auto-discovery.
- For **Agent-to-Agent / MCP**, treat dependencies between agents as the primary governance concern, capture the agent topology in Enterprise Architecture, and align governance with API Insights since MCP acts as a gateway/integration technology.
